## Core Principle: One Product, No Free/Pro Confusion

v0.5 makes the product easier to understand and use by removing the visible
Free/Pro split from the admin experience. The product direction is one AI Boost
for Joomla package with one paid license for updates and support, not a maze of
locked features, Pro badges, per-feature SKUs, and simulator states.

This does not mean deleting every compatibility hook in one pass. Legacy license
keys, historical `*_pro` package markers, and installer migrations may remain as
backward-compatibility shims until they are safely retired. The user-facing goal
is clear: admins should no longer feel they are using a crippled Free edition.

The single question every v0.5 decision should answer:
> **Does this make a normal Joomla admin less confused?**

If yes → v0.5 candidate.
If not → hold for v1 or post-v0.5.

## Key Decisions

### 1. One product, one license, no feature locking

**Decision:** AI Boost for Joomla is treated as one product. Per-feature SKUs are
retired, and visible Free/Pro feature locking is removed from the admin UI. The
paid license is for updates, support, and commercial entitlement, not for making
individual settings appear/disappear inside the component.

**Rationale:** the old Free/Pro and per-SKU model made users guess what they
needed and made the product feel fragmented. One product removes the cognitive
load. Lemon Squeezy / the update server can still enforce paid access to updates
and support; the component itself should not advertise a locked-feature maze.

**Impact on v0.5 work:** remove Pro badges, lock icons, muted Pro previews,
per-feature SKU language, and user-facing Free/Pro simulator affordances from the
admin experience. Keep legacy stored setting keys and migration paths stable
while the code is simplified in slices.

### 2. Simple-first navigation (v0.5 target menu)

**Decision:** the admin sidebar for v0.5 should be:

```text
OVERVIEW
- Dashboard
- Health          (absorbs Errors — one status surface, not two)

SETUP
- Autopilot       (new — guided one-time setup)
- Site Identity
- License & Updates
- Integrations

SEO
- Technical SEO   (canonical URL + 404 monitoring, moved out of Sitemap)
- Schema.org
- Sitemap         (sitemap-only, canonical/404 moved out)
- Social Meta
- Analytics & Tracking

AI VISIBILITY
- AEO             (llms.txt, Markdown endpoint, AI Signals, IndexNow)
- GEO             (post-v0.5 placeholder — do not implement in v0.5)
- Crawlers & Robots

TOOLS
- Redirects
- Analyzers
- URL Checker

ADVANCED
- Custom Code
- Debug
- Import
- Help
```

**What changed vs the old menu:**
- `Errors` merged into `Health` — one status surface
- `Autopilot` added as the first SETUP item (new feature, see §4)
- `General` removed from SETUP — domain/conflict settings stay accessible but
  are not a primary onboarding destination; they live inside relevant pages
- `License & Plan` → `License & Updates` — reflects one product, updates/support entitlement
- `AI SEARCH` → `AI VISIBILITY` — broader framing (not just search engines)
- `AI Search / AEO` → `AEO` — shorter, cleaner
- `GEO` listed but **not implemented in v0.5** — placeholder reserved
- `Technical SEO` is first in SEO group (most admins need it early)
- `Import` moved to ADVANCED — migration-oriented, not a normal workflow
- `Custom Code` and `Debug` in ADVANCED — developer tools, not daily SEO

**Backwards compatibility:** old query tab aliases (`tab=org`, `tab=aeo`, `tab=social`,
`tab=analytics`, `tab=sitemap`, `tab=code`) must remain working. Visible labels and
sidebar ordering change; stored setting keys do not change.

**Architecture gate:** Admin IA Gate 1 is already written in
`docs/architecture-refactor-plan.md`. Implement in slices as defined there.

### 3. Autopilot MVP (v0.5 scope)

**Decision:** add an Autopilot page as the first item under SETUP. The MVP scope:

- Guided one-time setup flow: Site Identity → Schema.org core → Sitemap → Social Meta
- Visual completion state (which key areas are configured)
- No AI backend in v0.5; Autopilot is a guided checklist, not an inference engine
- Designed to replace the current "new install confusion" where admins land on
  Dashboard and don't know where to start

**What Autopilot is not in v0.5:**
- Not an AI inference engine
- Not a replace for manual configuration
- Not a cron-based automation
- Not a "one-click optimize" button (post-v0.5)

**Architecture:** Autopilot is a new Vue route with read-only manifest inspection.
It does not require new PHP services in v0.5. Gate 0 required before implementation.

### 4. Health as the primary feedback surface

**Decision:** Health (merged with Errors) is the main way users understand what
needs attention. v0.5 should make Health more prominent and its fix actions more
accurate after the Admin IA refactor moves pages around.

**Specific v0.5 Health work:**
- Merge `Errors` into `Health` view (one status tab)
- Update Health fix target links after Admin IA navigation changes land
- Do not add new Health check categories in v0.5 unless required by a new feature

### 5. AEO/GEO framing: AI Visibility

**Decision:** rebrand the `AI SEARCH` section to `AI VISIBILITY`. This is a label
change only. The AEO page keeps its existing features (llms.txt, Markdown endpoint,
AI Signals, IndexNow). GEO is reserved as a future page but is not implemented in v0.5.

**Rationale:** "AI Visibility" better captures what the product does for AI engines
(ChatGPT, Perplexity, Google AI Overview, Bing Copilot) and aligns with the
aiboostnow.com marketing narrative.

### 6. Advanced options are secondary, not hidden

**Decision:** Custom Code, Debug, and Import move to ADVANCED. They remain accessible
to power users but are no longer in the primary navigation flow. This reduces noise
for the 90% of users who never touch them.

### 7. Free/Pro removal: staged, compatibility-first

**Decision:** v0.5 does touch the Free/Pro surface, but in slices. First remove
visible admin UI locking and per-SKU messaging. Then simplify server-side save
gating, registry exposure, simulator data, build split assumptions, and legacy
Pro package compatibility only where tests and staging can prove the change.

Do not delete backward-compatibility fields just because the UI no longer uses
them. Existing installs may still carry `license_tier`, `license_state`,
`pro_activated`, `pro_skus`, or installed `*_pro` extension rows. Those can remain
readable during the transition, but they must stop driving visible feature locks.

### 8. WordPress: post-v0.5

**Decision:** the WordPress vertical slice is deferred until after v0.5 ships.
Reason: shared service boundaries (especially `AiBoostCore.php` and settings save)
need to be stable before adding a second platform. Starting WordPress now would
create parallel duplication debt.

**Gate:** WordPress Gate 1 requires XHigh review. Do not start it during v0.5 work.

## v0.5 Release Sequence (recommended)

1. **Documentation & strategy** (this file, OPERATING.md, BACKLOG.md update) — ✅ done
2. **Free/Pro removal slice 1** — remove visible Pro locks/badges from sidebar, routes, page shells, and admin copy
3. **Free/Pro removal slice 2** — simplify save gating and registry exposure while preserving legacy settings compatibility
4. **Free/Pro removal slice 3** — retire per-SKU simulator/admin surfaces and align License & Updates with one product
5. **Admin IA slice 1** — sidebar grouping + visible labels + route aliases
6. **Admin IA slice 2** — Technical SEO page (canonical + 404 monitoring)
7. **Admin IA slice 3** — Crawlers & Robots page
8. **Health merge** — merge Errors into Health, update fix targets
9. **Autopilot MVP** — new SETUP route, guided checklist
10. **Admin IA slice 4** — remaining page moves (Meta Pixel, Social Meta, Schema.org order, AEO rename)
11. **QA pass** — full staging verification on all 4 test sites
12. **Release**

Steps 2–7 can be done in any order by separate agents as long as they don't collide
on the same files. Steps 2–4 are the highest-value UX changes and should ship first.
